# Deployment — Render via GitHub

How to put this app live on Render with auto-deploy from GitHub.

## Architecture in production

```
        GitHub (main branch)
              │ push
              ▼
       ┌─────────────────┐         ┌──────────────────────────┐
       │  Render builds  │ ──HTTP──┤ HuggingFace Space        │
       │  Docker image,  │         │ ibrahimdaud/text-embding │
       │  runs Gunicorn  │ ──SQL──▶│ Supabase Postgres        │
       │  (this repo)    │         │  + pgvector              │
       └─────────────────┘         └──────────────────────────┘
              │
              ▼
        users.onrender.com
```

The Render service hosts only Django. Postgres lives on Supabase
(already shared). Embeddings live on the HuggingFace Space (already
shared). Render is a stateless container.

## Files in this repo that drive the deploy

| File | Role |
|---|---|
| `Dockerfile` | python:3.12-slim, installs `requirements.txt`, runs `collectstatic` at build, starts Gunicorn on `$PORT`. |
| `.dockerignore` | Keeps the build context lean — excludes `.git`, `db.sqlite3`, `logs/`, `.env`, etc. |
| `render.yaml` | Declares the Render service: Docker runtime, autoDeploy, healthCheckPath, preDeployCommand, env-var template. |

You don't need to touch any of these for a normal deploy.

## One-time setup

### 1. Push to GitHub

The repo must be on GitHub (Render only deploys from GitHub / GitLab / Bitbucket). Make sure `Dockerfile`, `render.yaml`, and the rest of the repo are committed and pushed to `main`.

### 2. Create the Render Blueprint

1. Sign in to https://dashboard.render.com
2. Click **New → Blueprint**
3. Connect the GitHub repo (you'll authorise Render once)
4. Render reads `render.yaml` and shows you the service it's about to create:
   - Name: `eduai-platform`
   - Type: Web Service
   - Runtime: Docker
   - Plan: Starter ($7/mo, covered by your credits)
   - Region: Frankfurt (close to Supabase eu-west-1)
   - Health check: `/health/`
5. Click **Apply** to create the Blueprint.

### 3. Set the secret env vars

Render leaves any env var marked `sync: false` blank for security. Open the new service → **Environment** tab → set:

| Var | Value |
|---|---|
| `DATABASE_URL` | Your Supabase **transaction-pooler** URL (port `6543`). The same one in your local `.env`. |
| `EMBEDDER_API_KEY` | The 43-char token that gates the HuggingFace Space. |

`DJANGO_SECRET_KEY` is auto-generated by Render (no action needed thanks to `generateValue: true` in `render.yaml`). Other public env vars (model name, embedder URL, etc.) come pre-filled from `render.yaml`.

### 4. Trigger the first deploy

Render auto-builds on push, but for the very first deploy you can click **Manual Deploy → Deploy latest commit**. The build pipeline:

```
1. clone the repo
2. docker build -f Dockerfile .          # ~3-5 min first time
3. preDeployCommand: python manage.py migrate --noinput
4. start container with the CMD from Dockerfile
5. wait for /health/ to return 200      ← gates traffic switching
6. mark service Live
```

Watch the **Logs** tab until you see:

```
[INFO] Listening at: http://0.0.0.0:10000
INFO apps Applied 8 AppSetting override(s) to django.conf.settings.
```

That means the Dockerfile started Gunicorn and our AppConfig.ready() hook successfully pulled settings from the Supabase DB.

### 5. Verify the deploy

```bash
# Replace with your actual Render URL (visible in service dashboard)
curl https://eduai-platform.onrender.com/health/
# → {"status": "healthy", "checks": {"database": {"healthy": true, ...}}, ...}
```

## Day-to-day workflow

### Auto-deploy on push

```bash
git push origin main
```

Render watches the branch, rebuilds, runs migrations, swaps containers atomically. The whole cycle is 4-7 minutes.

### Run a one-off command (e.g. seed, bootstrap, custom shell)

Render service dashboard → **Shell** tab → opens a terminal inside the running container. Try:

```bash
python manage.py shell
python manage.py bootstrap_app_settings        # idempotent
python manage.py create_system_admin           # interactive
```

This is also how you would re-run `seed_synthetic_data` if you ever wipe the demo data.

### Apply a new migration (when you generate one)

`render.yaml` already runs `migrate --noinput` as `preDeployCommand`, so a normal `git push` is sufficient — the new migration applies before traffic switches over.

If a migration is risky (drops a column, big data backfill), gate the deploy:
1. Set `autoDeploy: false` in `render.yaml`
2. Push the migration in its own commit
3. Manual-deploy from Render dashboard
4. Verify, then re-enable autoDeploy

### Rotate the OpenAI key (or any AppSetting)

Don't redeploy. Just:

1. Go to `https://eduai-platform.onrender.com/admin/core/appsetting/`
2. Edit the row, paste new value, save
3. **Render service dashboard → Manual Deploy → Clear build cache & deploy** (or just Deploy latest commit) to restart the container so `ready()` picks up the new value

Skipping the restart leaves Django on the old in-memory value.

## Gotchas

### The free tier auto-suspends

Render's **Free** plan suspends after 15 min idle and takes ~30s to wake. The `Starter` plan ($7/mo, on your credits) keeps the service warm.

### Health-check failures on first boot

If `/health/` returns 503 right after a deploy, it almost always means:

| Symptom | Likely cause | Fix |
|---|---|---|
| `database: unhealthy` | `DATABASE_URL` wrong / Supabase transaction-pooler hit conn limit | re-paste URL (port 6543), check Supabase dashboard for active connections |
| `Application failed to respond` | `RENDER_EXTERNAL_HOSTNAME` not in `ALLOWED_HOSTS` | already auto-handled in `settings.py`; if you see this, double-check you didn't override `DJANGO_ALLOWED_HOSTS` to a too-narrow list |
| `502 Bad Gateway` | Gunicorn crashed | Render's Logs tab shows the traceback; usually a missing migration or env var |

### Memory pressure on the Starter plan

Starter has 512 MB RAM. Two Gunicorn workers consume ~250-300 MB plus model startup. If you start adding heavy work (long-running ingestion jobs), drop workers to 1 in `Dockerfile` `CMD`, or upgrade to Standard ($25/mo).

### Migrations through transaction pooler

`render.yaml`'s `preDeployCommand` runs against `DATABASE_URL` (transaction pooler, port 6543). For our small schema this works fine. If you ever hit a migration that fails with cryptic pgbouncer errors:

1. Set a second env var `DATABASE_URL_SESSION` to the session-pooler URL (port 5432)
2. Change `preDeployCommand` to:
   ```
   sh -c "DATABASE_URL=$DATABASE_URL_SESSION python manage.py migrate --noinput"
   ```
3. Push, deploy, then revert if you want.

This is a very rare path.

## Adding observability later

Three small env vars that, once set, light up production-grade monitoring without code changes:

| Env var | What it enables |
|---|---|
| `SENTRY_DSN` | Error reporting via Sentry (already wired in `settings.py`) |
| `REDIS_URL` | Replaces the in-memory cache with Redis (faster, multi-worker) |
| `LOG_FILE` | Path inside the container (default `logs/app.log`) — point at a Render disk if you add one |

Add them to `render.yaml` (uncomment the production-only block) or set in the dashboard.

## Rolling back

Render keeps every deploy. Service dashboard → **Deploys** tab → click an old commit → **Rollback to this deploy**. ~30s to swap. Same DB; only the container changes.

## Tearing it down

Service dashboard → **Settings** → **Delete Service**. Confirms with a typed name. Doesn't touch Supabase or the HF Space — those are separate.
